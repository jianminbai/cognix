## 模板 1：ES 日志异常分析

### 🎯 适用场景
收到告警后，从 ES 日志中快速定位根因。

### 📝 模板

```
你是一个资深 SRE 工程师。分析以下 Elasticsearch 日志片段：

[在此粘贴 Kibana 搜索到的异常日志，10-30 行即可]

系统环境：
- 服务名称：[服务名]
- 告警内容：[告警原文]
- 时间范围：最近 [X] 分钟
- 指标情况：[CPU/内存/延迟等]

请按以下格式输出：
1. 问题分类（资源 / 应用 / 依赖 / 外部）
2. 最可能根因 Top 3（按概率从高到低排列）
3. 每条根因对应的排查命令
4. 下一步建议

禁止输出空泛建议。排查命令必须是具体可执行的，比如 curl、kubectl、jstack 等。
```

### 💡 技巧
日志行数控制在 10-30 行，太多 AI 会丢失焦点。只贴关键错误行和上下文。

## 模板 2：Prometheus 告警根因分析

### 🎯 适用场景
Prometheus 告警触发，需要快速判断是"真故障"还是"误报"。

### 📝 模板

```
分析以下 Prometheus 告警：

告警名称：[alertname]
标签：{[labels]}
指标值：[当前值 / 阈值]
持续时间：[duration]

相关服务：[service]
近期变更：最近 [X] 小时内是否有部署/配置变更

请判断：
1. 这是真故障还是误报？依据是什么？
2. 如果是真故障，Top 2 可能原因
3. 是否需要立即介入处理（Y/N）
4. 如果不需要立即处理，建议观察什么指标恢复到什么值
```

### 💡 技巧
对于 CPU/内存类告警，加上"对比昨天同时间的值"——AI 擅长对比趋势。

## 模板 3：SkyWalking 慢调用链路分析

### 🎯 适用场景
SkyWalking 发现某个接口延迟飙升，需要定位瓶颈在哪一环。

### 📝 模板

```
分析以下 SkyWalking Trace 数据：

端点：[endpoint]
P95 延迟：从 [正常值] 升到 [当前值]
调用链：
[贴 Trace 瀑布图的关键 Span 信息，包含各服务的耗时]

请分析：
1. 延迟最长的服务/方法是什么
2. 是自身慢还是下游慢（自身慢 = 代码/DB/GC；下游慢 = 依赖服务）
3. 建议检查的方向（具体到某个服务或方法）
4. 同时期日志中有无对应异常
```

### 💡 技巧
Zipkin/SkyWalking 的瀑布图文本化后粘贴给 AI。关键信息：Span 名称 + 耗时 + 是否有 Error。

## 模板 4：Zabbix 告警降噪判断

### 🎯 适用场景
同一故障触发多个 Zabbix 告警，需要合并并判断真正的根因。

### 📝 模板

```
收到以下 Zabbix 告警列表（最近 [X] 分钟内）：

[列出告警标题和时间，每行一个]

请分析：
1. 哪些告警可能是同一根因触发的？分为几组
2. 每组告警的根因是什么
3. 哪些告警是"可直接忽略的从属告警"
4. 哪些是真正的 P0/P1 告警需要立即处理
5. 这条告警规则是否可以优化（是否有持续误报）
```

## 模板 5：Docker/K8s 容器异常排查

### 🎯 适用场景
容器频繁重启、CrashLoopBackOff、OOMKilled。

### 📝 模板

```
分析以下容器异常：

Pod 状态：[CrashLoopBackOff / OOMKilled / Error]
描述：kubectl describe pod 中的 Last State 和 Exit Code
日志最后 20 行：[贴日志]

系统配置：
- 内存限制：[数值]
- CPU 限制：[数值]
- 健康检查：[liveness/readiness 配置]

请分析：
1. 重启原因（OOM / 应用崩溃 / 健康检查失败）
2. 如果是 OOM，是内存泄漏还是限制太小
3. 修复建议（具体到调整什么参数）
```

## 模板 6：故障时间线复盘

### 🎯 适用场景
故障恢复后做事故复盘（Postmortem）。

### 📝 模板

```
帮我整理以下故障事件的时间线并生成复盘报告：

故障描述：[一句话描述]
影响范围：[影响的服务 / 用户]
时间线（按顺序排列）：
[T1] [事件1]
[T2] [事件2]
[T3] [事件3]
...

请输出：
1. 标准时间线（时间 → 事件 → 影响）
2. MTTR（从告警到恢复的时长）
3. 根因分类（人为 / 系统 / 流量 / 依赖）
4. 可改进项（监控 / 告警 / 流程 / 代码）
5. 建议的 Action Item（谁 + 何时完成）
```

## 模板 7：告警规则健康检查

### 🎯 适用场景
定期审查你的 Prometheus/Zabbix 告警规则，减少误报和漏报。

### 📝 模板

```
审查以下告警规则：

[列出 5-10 条告警规则，包括：规则名、表达式、阈值、持续时间]

对于每条规则，请判断：
1. 这条规则在过去一周产生过多少告警
2. 其中多少是真实故障、多少是误报
3. 阈值是否合理（偏高/偏低）
4. 是否需要调整（新阈值建议 + 理由）
5. 这条规则是否可以删除或合并
```

## 模板 8：Java 应用日志分析（针对 OTel 采集）

### 🎯 适用场景
OTel Collector 采集的 Java 应用日志，包含自定义 regex 解析后的结构化日志。

### 📝 模板

```
分析以下 OpenTelemetry Collector 采集的 Java 应用日志：

[粘贴 10-20 行结构化日志，包含时间戳、级别、类名、消息]

日志格式说明：
- 采集方式：filelog receiver
- 解析方式：[regex 或 json 解析器]
- 应用类型：Java Spring Boot / 其他

请分析：
1. 日志异常模式分类
2. 最常见的错误类型及其占比
3. 哪些错误是已知的"背景噪音"
4. 建议新增什么告警规则来捕获真正需要关注的问题
5. 建议忽略哪些重复错误以减少日志存储
```

## 模板 9：CI/CD 流水线失败分析

### 🎯 适用场景
GitHub Actions / GitLab CI 流水线失败，需要快速定位原因。

### 📝 模板

```
分析以下 CI/CD 流水线失败：

流水线类型：[GitHub Actions / GitLab CI / Jenkins]
失败阶段：[build / test / deploy / scan]
错误日志最后 20 行：[粘贴]

最近变更：[最近一次提交改了哪些文件]

请分析：
1. 失败根因
2. 是代码问题还是环境问题
3. 修复方案
4. 如何避免同类问题再次发生（加测试 / 加检查）
```

## 模板 10：多工具联合排查（ES + Prometheus + SkyWalking）

### 🎯 适用场景
复杂故障需要同时看日志、指标和调用链。

### 📝 模板

```
我有同一个故障的三个数据源，请做联合分析：

ES 日志摘要：[3-5 行关键日志]
Prometheus 指标变化：[异常指标 + 时间点]
SkyWalking Trace 摘要：[慢调用或错误 Span]

问题现象：[一句话描述]

请做交叉分析：
1. 三个数据源各自指向什么方向
2. 是否存在矛盾信号（比如日志说 A，指标说 B）
3. 综合判断最可能的根因
4. 下一步最应该查什么（从哪个工具哪个角度入手）
```
